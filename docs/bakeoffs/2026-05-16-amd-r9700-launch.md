# Bakeoff — 2026-05-16 — Inaugural run on AMD Radeon AI PRO R9700

**Trigger**: First bakeoff after migrating from Nvidia RTX 3080 (10 GiB) to AMD
Radeon AI PRO R9700 (32 GiB). Goal: validate that local 30B-class models on the
new GPU are competitive enough with Claude Code for the workflows this server
actually runs (n8n, imperium-loop, ad-hoc CLI).

**Hardware**: AMD Radeon AI PRO R9700, ROCm 6.4.4, Ollama HEAD as of 2026-05-16
**Generators**:
- `claude` (Opus 4.7 via `claude -p --output-format json`, tools disabled)
- `gemma4:31b` (Google, dense, 30.7B params, 256K context)
- `gemma4:26b` (Google, MoE, 25.2B total / 3.8B active, 256K context)
- `gemma4:e4b` (Google, 4.5B effective, 128K context — anchors the floor)
- `gpt-oss:20b` (OpenAI, MoE, 20.9B total / 3.6B active, 131K context — added
  after the initial run to put a non-Google open-weights model in the comparison)

**Judges (all three saw the same 5-answer batch per prompt)**:
- claude-opus-4-7 (me, interactive — read every response)
- gemma4:26b (`/api/chat` with `think: false`)
- gpt-oss:20b (`/api/chat` with `reasoning_effort: "low"` + `system: "Reasoning: low"`)

## TL;DR

| Generator | My GPA | 26b GPA | gpt-oss GPA | Per-task cost | Wall-clock |
|---|---|---|---|---|---|
| **claude** | **4.00** | **3.88** | **3.63** | **$0.11 avg** | 31s |
| gemma4:31b | 3.53 | 3.75 | 3.25 | $0 marginal | 87s |
| gpt-oss:20b | 1.88 | 3.13 | 1.88 (self) | $0 marginal | 22s |
| gemma4:26b | 1.63 | 2.88 (self) | 1.50 | $0 marginal | 43s |
| gemma4:e4b | 1.30 | 2.00 | 0.88 | $0 marginal | 28s |

**All three judges agree on the rank order**: claude > 31b > gpt-oss > 26b > e4b.
The disagreement is on *spread* — gemma4:26b is meaningfully lenient on
compile-blocking bugs (closes the gap by ~30%); gpt-oss is harsher (especially
on details like missing type annotations) and lands closest to my read.

**Headline**: gemma4:31b lands at ~88% of Claude's quality (3.53 / 4.00 GPA)
for $0 marginal cost. For batch or autonomous workloads where Claude's tool use
isn't load-bearing, that's the win we hoped for from the GPU swap. For one-off
questions that fit in a chat box, Claude is still worth the dime.

**Surprise**: `gpt-oss:20b` (OpenAI's open-weights MoE, ~3.6B active params,
explicitly trained for tool-calling) **does NOT outperform gemma4:31b in raw
text-gen mode** — and lands close to gemma4:26b (also an MoE) in failure rate.
Different failure modes (hallucinated APIs vs typos / duplicate fields) but
similar compile-blocking density. Tool training doesn't pay back when you're
not actually using tools.

**Caveat**: the benchmark deliberately disabled Claude Code's tools. Real-world
Claude advantage — Read/Edit/Bash/Grep/Agent on the actual codebase — does not
show up here. Treat these grades as "how well can the underlying model answer
a coding question in one shot."

## Cost framing

Claude charges per-call. Each cold invocation pays a **~16K-token
cache-creation tax** (system prompt + project context) — that's `claude -p`
overhead, not your prompt. Within a 5-minute window the cache reads cheaply;
the bakeoff fired calls sequentially so most were cache-warm.

| Tier | Practical capacity at $0.11/task avg | What that buys |
|---|---|---|
| Pay-as-you-go API | $1 → ~9 tasks | Marginal, fair pricing |
| Pro ($20/mo) | ~180 tasks if linear (rate limits hit first) | Casual daily use |
| Max ($200/mo) | ~1800 tasks if linear | ~60/day, heavy daily driver |
| Local (gemma4:31b) | **unlimited** at ~$0.0001/task electricity | 24/7 batch / agent loops |

**Where local wins**: background agents (imperium-loop autonomous runs), bulk
review / doc generation / ticket triage, cheap idea generation where 88%
quality is fine, privacy-sensitive contexts.

**Where Claude wins**: anything tool-augmented (not measured here), one-off
complex tasks where your time cost dominates the API bill, strict-correctness
work where the 12% delta matters (migrations, security, critical refactors).

**Where the judges diverge from me — and why**:

- **Gemma is the most lenient judge**, especially on the bottom-tier models.
  Same finding as the original 4-generator pass: gemma reads structure and
  commentary, doesn't simulate the toolchain. It graded gpt-oss's broken Go
  handler (`pgx.FormatError`, unused imports) an A. It graded gemma4:26b's
  hallucinated `"und"` enum a C instead of D/F.
- **Gpt-oss is harsher than gemma, closer to my read**, but with two oddities:
  1. **Prompt 05 (Hono+Zod): gave everyone D except itself an A**, claiming
     the others "fail to use typed `c.json<Ticket>()`" — which is false; both
     claude and gemma4:31b use it. Hallucinated critique combined with self-bias.
  2. **Prompt 06 (Drizzle): gave EVERYONE D including itself** — overcorrection.
     Caught real bugs, but flagged claude's correct schema with a contrived
     critique. Better-than-gemma at finding bugs, worse-than-gemma at giving
     credit when nothing is wrong.
- **Gpt-oss self-graded F on prompt 03 (tests)** — correctly caught its own
  bad expectation for "multiple hyphens" cases. Better self-judgment than
  gemma4:26b ever showed.

**Where all three judges agreed strongly**: rank-order claude > 31b > gpt-oss
> 26b > e4b. Claude consistently A or near-A. Bottom of the table consistently
near-failing. The middle (gpt-oss vs 26b) was a coin-flip across judges.

## How each model failed when it did

**gemma4:31b** — failures were *omissions*, not bugs:
- Sometimes missed one bug in a multi-bug prompt (4/5 vs 5/5 on concurrency)
- Sometimes left out idempotency in migrations (forgot delivery-id dedup)
- Less awareness of edge cases (didn't proactively flag Sscanf trailing-input
  behavior the way claude did)
- Code itself always compiled and ran correctly

**gpt-oss:20b (OpenAI MoE)** — failures were *confident hallucinations*:
- `pgx.FormatError(err)` — invented API, not in pgx v5 or any earlier version
- Unused imports `errors` and `context` while still claiming to be idiomatic
  Go (won't compile)
- Dropped the `sync` import from a corrected concurrency fix while still
  referencing `sync.WaitGroup`
- Stated ALTER TABLE takes an `ACCESS SHARE` lock that "blocks writes" — both
  halves wrong; the prompt specifically asked about Postgres locking
- SQL `WHERE next_attempt_at IS NULL AND next_attempt_at <= now()` — the
  second predicate evaluates to NULL whenever the first is true, so the
  worker would never claim a job
- For the async-refactor prompt: identified the null-pointer bugs correctly,
  then **explicitly preserved them** with non-null assertions and called it
  "preserving semantics" (the prompt said *fix* the bugs)

Pattern: gpt-oss is fluent and well-organized at the structural level but
generates compile-blocking detail errors at a higher rate than gemma4:31b.
Counterintuitive given OpenAI specifically trained it for tool-calling /
reasoning loops — but in pure text-gen mode that training doesn't help, and
may actively hurt: gpt-oss confidently emits APIs that "should exist."

**gemma4:26b (Google MoE)** — failures were *bugs*:
- `"Content/Type"` instead of `Content-Type` in HTTP headers
- Typo `ticketlyId` instead of `ticketId` (TypeScript compile error)
- Hallucinated `"und"` value in a pgEnum
- Duplicate struct fields in Go test cases
- Calling `.errorMap()` as if it were a method on Zod schemas (it isn't)

Pattern: the MoE routing (3.8B of 25.2B params active per token) is good at
"general shape of an answer" but weak on detail-level correctness. **Similar
overall GPA to gpt-oss but different failure mode** — gpt-oss invents APIs
with confidence, gemma4:26b makes typos and copy-paste errors.

**gemma4:e4b** — failures were *structural*:
- Introduced a deadlock in concurrent code (unbuffered channel send before
  wg.Done)
- Defined mock types/functions inline when the prompt told it to import them
- Missed parallelization requirements that were stated explicitly
- Hallucinated columns (referenced `updated_at` in failure-mode analysis after
  defining a schema without it)

## Token economy

Per-task averages across 8 prompts:

| Generator | Avg input tokens | Avg output tokens | Avg cost (USD) |
|---|---|---|---|
| claude | 16,386 (mostly system-prompt cache) | 2,387 | $0.11 |
| gemma4:31b | 411 | 1,904 | $0 marginal |
| gemma4:26b | 411 | 2,536 | $0 marginal |
| gemma4:e4b | 411 | 1,750 | $0 marginal |
| gpt-oss:20b | 427 | 2,372 | $0 marginal |

Claude's ~16K input/call is the per-invocation system-prompt cost. Inside a
long Claude Code session it gets cached; firing `claude -p` from a script (as
this bakeoff did) pays full freight each time. Heavy batch workloads should
run inside a single interactive session — per-task cost drops by ~80%.

**`gpt-oss:20b` quirk on tokens**: the `eval_count` reported by Ollama
significantly exceeds the visible character output. Example: prompt 03 emitted
4,321 tokens to produce a 1,876-byte file. The harmony format's hidden
reasoning channel accounts for the gap — Ollama strips reasoning from the
visible response but it still spends generation budget. The same effect made
batched judging fail until we capped reasoning with `reasoning_effort: "low"`
plus a `Reasoning: low` system message AND bumped `num_predict` to 4000
(8000 for the longest prompts). See [the gotchas section](README.md#gotchas)
of the bakeoff README.

## Workflow recommendations

1. **Daily coding work** (interactive, file edits): **Claude Code**. Tools
   matter more than raw model quality.
2. **n8n / autonomous loops that call an LLM**: **gemma4:31b** via Ollama.
   88% of the quality for $0; perfect for fan-out workloads. **Stays the
   recommendation over gpt-oss:20b** despite gpt-oss being newer — gpt-oss's
   3.6B active params hit the same MoE detail-loss wall as gemma4:26b.
3. **Quick "what does this code do" lookups from CLI**: either claude or
   gpt-oss:20b. gpt-oss is the fastest local option (~22s/task vs gemma4:31b's
   87s) — when you want a fast read on something but don't need the answer to
   compile, gpt-oss is fine.
4. **Code review on small PRs**: gemma4:31b is a useful second opinion. **Do
   not trust gemma4:26b OR gpt-oss:20b's judgment on compile-level
   correctness** — both eyeball structure rather than simulating the toolchain,
   though gpt-oss is slightly better at catching its own family's bugs.
5. **e4b**: utility model only (fast classification, simple text gen). Not
   viable for codegen.

### Tool-augmented follow-up (TODO)

This bakeoff measured raw text gen. The most actionable next experiment is
re-running the same 8 prompts through a **tool-augmented agent loop** that
gives each local model the same `Read` / `Bash` / `Edit` affordances Claude
Code has — specifically a "run the linter / typechecker / tests, iterate on
errors" cycle. Hypothesis: tool-augmented gemma4:31b closes most of the 12%
quality gap to Claude; tool-augmented gpt-oss:20b (whose training emphasizes
tool calling) **may even outperform** non-augmented Claude on prompts where
the bug pattern is detectable by `tsc --noEmit` / `go vet`. Worth knowing
before committing to a specific agent harness.

## What changed since the previous setup

Previous: RTX 3080 (10 GiB) running `gemma3:12b` as the daily driver.

After this swap:
- 32 GiB VRAM unlocks `gemma4:31b` (20 GB) as primary, with `gemma4:26b`
  (17 GB), `gpt-oss:20b` (13 GB), and `gemma4:e4b` (10 GB) as alternates
- ROCm 6.4.4 has native `gfx1201` support — no `HSA_OVERRIDE_GFX_VERSION`
  needed
- Migration to AMD removed the Nvidia DKMS / Container Toolkit / Datadog GPU
  reservation path; Ollama now uses `/dev/kfd` + `/dev/dri` device passthrough
- No previous bakeoff exists to A/B against — this is the baseline future
  runs will compare to
