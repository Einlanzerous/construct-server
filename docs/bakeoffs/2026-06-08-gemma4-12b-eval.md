# Bakeoff: gemma4:12b evaluation

**Date:** 2026-06-08
**Trigger:** New `gemma4:12b` (7.6 GB) released; question is whether it should
join or displace `gemma4:31b` as the local coding generator.
**Run:** `results/20260608-141122/` — 8 prompts × 5 generators = 40 generations.
**Lineup:** claude (reference) · gemma4:12b · gemma4:31b · gemma4:e4b · qwen3-coder:flash
**Judges:** Claude session (interactive) + gemma4:31b (local, batched). Neither saw the other's grades.

## Result

| Generator | Claude-judge GPA | Local-judge GPA | avg latency | avg output tok | Notes |
|---|---|---|---|---|---|
| claude | **4.00** | **4.00** | 40.9s | 3200 | Reference. $1.11 total for the 8 prompts. |
| gemma4:31b | **3.62** | **3.38** | 93.8s | 1639 | Still the local champ. Tight, mostly-compiling answers. |
| **gemma4:12b** | **2.00** | **2.25** | **83.8s** | **3360** | New model. Mid-pack. See below. |
| qwen3-coder:flash | 1.88 | 2.00 | 23.6s | 650 | Fastest, but terse to a fault; misses requirements. |
| gemma4:e4b | 1.50 | 1.50 | 34.3s | 1758 | Floor anchor. Hallucinated APIs, broken migrations. |

Both judges produced the **identical ranking**. GPA scale A=4…F=0, graded absolutely.

## Verdict: gemma4:12b does **not** displace gemma4:31b

`gemma4:12b` lands a clear tier below the 31b (2.0 vs 3.6) and only marginally
ahead of the much smaller/faster alternatives. It should **not** become the
default local coding generator. `gemma4:31b` stays the recommendation.

### Signature failure mode: "almost right, every time"

The 12b was remarkably *consistent* — the Claude judge gave it a **C on all 8
prompts** (never an A, never a D/F). Every answer had logically-correct
structure undone by exactly **one** disqualifying defect:

- **01** broken import path `github./jackc/...` + unused imports → won't compile
- **03** asserts `SWY-12b` errors when `Sscanf` actually reads `12`
- **04** `currentSeverity < bestSeverity` compares against `number | undefined` → strict-TS compile error
- **05** uses bare `zValidator` with no failure hook → omits the required custom 400 body
- **06** uses `uuid('project_id')` but never imports `uuid` → won't compile
- **07** parallelizes only 2 of 3 fetches; invents a silent `{id:'unknown'}` fallback
- **08** no constant-time HMAC compare, no idempotency/delivery-id dedup

This is the inverse of e4b/qwen, which fail *loudly* (hallucinated APIs,
invalid SQL, F-grade collapses). The 12b is reliable but mediocre — it gets you
80% of the way and then trips on one compile error or one dropped requirement.

### The size advantage didn't show up as speed

The 7.6 GB footprint *should* make 12b fast, but it was **slower than expected
(83.8s avg, barely faster than the 19 GB 31b's 93.8s)** because it is the most
**verbose** local model — 3360 avg output tokens, more than any other local
model and even more than Claude. It reasons out loud and rambles, spending its
token-throughput advantage on length rather than latency.

Where the small footprint *would* pay off: 7.6 GB can **co-reside** with another
model in the R9700's 32 GB VRAM, avoiding the model-swap reloads that dominate
multi-model pipeline wall-clock. That's an operational win for pipelines that
juggle two models — but it's not a coding-quality argument.

## Recommendation

- **Keep `gemma4:31b`** as the default local generator (`CENTRIFUGE_OLLAMA_MODEL`, autonomous loops).
- **Consider `gemma4:12b` only** where its VRAM footprint enables co-residence
  with another model, and the task tolerates "mostly right" output (drafting,
  non-critical summarization) — not for code that has to compile first try.
- Skip it for the coding/autonomous-loop role; the quality gap vs 31b is real
  and consistent across two independent judges.

## Caveats

- Single run, n=8 per generator, default sampling — one data point, not a distribution.
- Wall-clock includes Ollama model-swap latency (no two ≥17 GB models co-reside in 32 GB),
  so per-generation seconds are confounded by reload order; treat latency as directional.
- "Won't compile" judgments are from reading, not from actually running `go build` / `tsc`.
